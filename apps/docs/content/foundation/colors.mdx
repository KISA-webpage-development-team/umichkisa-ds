# Colors

Color is one of the most visible decisions in any interface. Done inconsistently,
it creates visual noise — a button is one shade of blue here, another shade there,
a heading feels slightly off, and the page looks like it was assembled by different
people who never talked to each other. Because it was.

The KISA color system exists to solve that. It gives every developer on the team a
shared vocabulary: instead of choosing a color by eye or copying a hex value from
somewhere, you reach for a named token — `--color-brand-primary`,
`--color-text-muted` — and the right color appears, consistently, everywhere.

This guide covers how the color system is structured, the full palette we build
from, the tokens you use in components, and the rules that keep everything
consistent.

If you have never worked with a design system before, start at the top and read
through. If you already know what you are looking for, use the section links in
the sidebar.

## How the System Works

The color system is organized in three tiers. Each tier has a specific job,
and they only communicate in one direction — downward.

### Tier 1 — Primitives

Primitives are the raw material. They hold color values and nothing else —
no meaning, no context, just the value. You will find them in
`packages/web/src/tokens/primitives.css`.

You will never reference a primitive directly in a component. Their only job
is to be referenced by semantic tokens.

> **Why OKLCH?** Hex values (`#00274c`) are unreadable — you cannot tell anything
> about a color from the code alone. OKLCH is perceptually uniform, meaning a 10%
> increase in lightness actually looks 10% lighter to the human eye, making the
> palette predictable. Reading an OKLCH value: `oklch(lightness% chroma hue)`.

### Tier 2 — Semantic Tokens

Semantic tokens are named by their purpose. This is the tier you interact with
when writing component code. You will find them in
`packages/web/src/tokens/semantic.css`.

```css
--color-brand-primary: var(--primitive-michigan-blue);
--color-text-primary:  var(--primitive-gray-900);
```

The name tells you what the color is *for*, not what it looks like. This matters
for maintainability: if KISA ever updates its brand colors, only the primitive
value changes. Every component that references the semantic token updates
automatically — without touching a single component file.

**Rule: always use semantic tokens in components. Never reference primitives directly.**

### Tier 3 — Component Layer

The component layer is where semantic tokens get applied to actual UI elements.

```css
.btn-primary {
  background-color: var(--color-brand-primary);
  color:            var(--color-text-on-brand);
}
```

The component does not know what color `--color-brand-primary` is. It only knows
the purpose. The actual value flows in from the semantic layer, which in turn
pulls it from primitives. This separation is what makes the system maintainable.

Background colors that define the depth layers of the UI. Think of them as

|---|---|---|

## Usage Guidelines

This section explains when to reach for each token. The token reference above
tells you what each token is. This section tells you how to use it.

### Brand Colors

**`--color-brand-primary` is the anchor. `--color-brand-accent` is the signal.**

Think of navy as the container and maize as the content inside it. Navy says
"this is KISA." Maize says "look at this."

Use `--color-brand-primary` for:
- Navigation bar background
- Hero section backgrounds
- Primary button backgrounds
- Any large surface that needs to feel institutional and grounded

Use `--color-brand-accent` for:
- Text labels on top of `--color-brand-primary` backgrounds
- Focus ring on interactive elements
- Badges, tags, and highlights that need to draw attention
- The most important CTA on a page when you want maximum visibility

Use `--color-brand-primary-mid` for:
- Hyperlinks within body text
- Secondary navigation highlights
- Any context where full navy would feel too heavy but you still need blue

Do not scatter brand colors across every element. A page that is half navy and
half maize is not more branded — it is just loud. Brand colors work because they
are used deliberately and sparingly.

### Surface Depth

Surfaces create the sense of depth and layering in the UI. The rule is simple:
as elements stack on top of each other, their background steps through the
surface scale.

```
--color-surface               (white)      Page background
  └── --color-surface-subtle  (gray-100)   Cards, panels
        └── --color-surface-muted  (gray-50)    Items inside cards
```

A practical example: the page sits on `--color-surface`. An event card sits on
`--color-surface-subtle`. Inside that card, individual detail rows use
`--color-surface-muted` — which is lighter than the card, lifting items visually
above the card background.

Do not skip levels. A card item should not sit directly on `--color-surface`
(the page background) — it would look uncontained. Do not invent new background
colors outside of these three tokens.

### Text Hierarchy

Use the text tokens to communicate information priority.

**`--color-text-primary`** — everything users need to read. Headings, body
paragraphs, button labels, form values. If it carries meaning, it gets primary.

**`--color-text-muted`** — supporting information. Timestamps, captions, helper
text below form fields, secondary metadata on cards. It is readable but visually
recedes behind primary text.

**`--color-text-disabled`** — text on disabled UI elements only. It is
intentionally below WCAG contrast requirements because disabled elements are
meant to signal "this is not available right now." Never use this token for
content you actually want someone to read.

**`--color-text-on-brand`** — the only text token to use when the background is
`--color-brand-primary`. On a dark navy surface, gray-900 text is nearly
invisible. Maize text is what belongs there. Do not use white — it is not a KISA
brand color and it disrupts the visual identity.

**`--color-text-link`** — hyperlinks only. It is a distinct blue that signals
clickability. Do not use it for decorative text. Do not use `--color-brand-accent`
(maize) for links — yellow text on a white background is nearly unreadable and
gives no visual signal that the text is interactive.

### Feedback Colors

Each feedback state has a solid token and a subtle token. Use them as a pair.

The **solid token** (`--color-error`, `--color-success`, etc.) goes on text,
icons, and borders — small, precise uses where the color itself carries the
message.

The **subtle token** (`--color-error-subtle`, `--color-success-subtle`, etc.)
goes on the background of the entire alert or callout box. It creates a tinted
region around the message without overwhelming the page.

A typical error alert uses both: `--color-error-subtle` as the box background,
`--color-error` as the icon color and border.

When to use each state:
- **Error** — something went wrong and the user must act. Form validation
  failures, failed submissions, destructive warnings.
- **Success** — an action completed as expected. A form saved, a file uploaded,
  a password changed.
- **Warning** — something needs attention but has not failed yet. A session about
  to expire, a form field with a recommendation, a soft limit approaching.
- **Info** — neutral context. Tips, documentation callouts, informational notices
  that require no action.

### Interactive States

Every interactive element that uses brand colors should cycle through these
states: default → hover → pressed → (focus).

For elements built on `--color-brand-primary`:

| State | Token |
|---|---|
| Default | `--color-brand-primary` |
| Hover | `--color-brand-primary-hover` |
| Pressed | `--color-brand-primary-pressed` |
| Focus ring | `--color-focus-ring` |

For elements built on `--color-brand-accent`:

| State | Token |
|---|---|
| Default | `--color-brand-accent` |
| Hover | `--color-brand-accent-hover` |
| Pressed | `--color-brand-accent-pressed` |
| Focus ring | `--color-focus-ring` |

The focus ring (`--color-focus-ring`) is always maize regardless of which brand
color the element uses. It appears when a user navigates with a keyboard instead
of a mouse. It must always be visible — do not remove it, do not reduce its
opacity, do not hide it with `outline: none`.

## Accessibility

Color contrast is the ratio between the brightness of text and the brightness of
its background. A ratio of 1:1 means no contrast at all — white text on white.
A ratio of 21:1 is the maximum — black text on white.

WCAG AA is the industry standard accessibility level. It requires:
- **4.5:1** minimum for normal body text
- **3:1** minimum for large text (18px and above, or 14px bold)
- **3:1** minimum for UI components such as button borders and input outlines

The table below shows the contrast ratios for the token pairs you will use most
often. Pairs labeled **AA** pass WCAG AA for normal text. Pairs labeled
**Large only** pass only for large text (18px+ or 14px bold). Pairs labeled
**By design** intentionally fail — they are designed that way.

**On `--color-text-disabled`:** The 2.8:1 ratio is intentional. Disabled
elements are not meant to be read — they communicate "this action is not
available." If a user needs to read it, it should not be disabled.

**On `--color-text-muted` over `--color-surface-subtle`:** This combination
passes at 4.2:1 for large text only. Avoid using muted text at small sizes
inside cards — use `--color-text-primary` instead if the content matters.

**On `--color-text-on-brand` over `--color-brand-primary`:** This is the
strongest verified pair in the system at 8.2:1 — maize on navy. It clears
WCAG AA at every text size and weight. Whenever text sits on a brand-primary
background, this is the combination to reach for. It is also the combination
that defines KISA's visual identity.

**On `--color-text-on-brand` over hover and pressed states:** Maize remains
valid on `--color-brand-primary-hover` (7.2:1) and `--color-brand-primary-pressed`
(10.3:1). Both clear WCAG AA. There is no need for a different text token on
interactive states.

**On feedback colors over `--color-surface`:** `--color-error` (3.9:1) and
`--color-warning` (3.0:1) pass for large text only — do not use them as small
body text. `--color-success` (2.2:1) and `--color-warning` fall below the
non-text contrast threshold of 3:1 on white backgrounds, so use them paired
with `--color-text-primary` labels for any readable content. `--color-info`
(5.9:1) passes for all text sizes.

    <pre><code>background: var(--color-surface-subtle)</code></pre>
  </Do>
  <Dont>
    <p>Use brand colors as mid-page backgrounds. A navy card mid-page does not feel branded — it feels heavy and breaks reading flow.</p>
    <pre><code>background: var(--color-brand-primary)</code></pre>
  </Dont>
</DoDont>

### Token usage in components

<DoDont>
  <Do>
    <p>Reference semantic tokens in all component CSS and inline styles. One token change updates the entire system.</p>
    <pre><code>background-color: var(--color-brand-primary)</code></pre>
  </Do>
  <Dont>
    <p>Hardcode hex values in components. A hardcoded value cannot be updated globally and drifts out of sync with the design system over time.</p>
    <pre><code>background-color: #00274c</code></pre>
  </Dont>
</DoDont>
